# Label-X — Part 5: Chatbot (MiniMax M3) & Translation (Gemini) PRD

## 1. Scope of This Document
This defines the quick-response chatbot feature (using MiniMax M3 via NVIDIA NIM) and
the language translation feature (using Gemini). Both are additions on top of the core
scan pipeline defined in Parts 1-3 — neither replaces or modifies the main scan flow.

## 2. Why Two Different Models (state this explicitly if asked by judges)
- **Gemini** handles the main scan pipeline (multimodal image + reasoning + FSSAI
  claim checking + QUID + personalization) and translation of final output text.
- **MiniMax M3 (via NVIDIA NIM)** handles the chatbot — a separate, lighter, faster
  path for users who don't want to wait through the full scan flow.
- This is an intentional architectural split to keep the system resilient and not
  bottlenecked on a single provider — not a workaround, a deliberate design choice.

## 3. Chatbot — Purpose and Behavior

**Purpose:** some users won't have patience for the full scan flow (camera capture →
loading state → full result page). The chatbot offers a quick, conversational
alternative: the user can just ask, in plain language, whether a food/ingredient/claim
is okay for them — and get a fast answer grounded in their health profile.

**Critical constraint:** the chatbot is NOT a replacement for the scan pipeline and
must never claim the same level of accuracy. It does not have access to actual product
images or nutrition panels unless the user describes them in the chat — it reasons
from whatever the user tells it, plus their stored health profile. Every chatbot
response must make this limitation clear (see §3.5).

### 3.1 Chatbot Route
| Route | Method | Purpose | Request Body | Response Body |
|---|---|---|---|---|
| `/api/chat` | POST | Send a message to the health chatbot | `{ userId, message, conversationHistory: [] }` | `{ reply, disclaimer }` |

- `conversationHistory` is an array of prior `{ role: "user" | "assistant", content: string }`
  turns from the current chat session, sent by the frontend each time (no server-side
  session state required for hackathon scope — keep it stateless per request, frontend
  manages the running conversation array).

### 3.2 Backend Flow for `/api/chat`
1. Receive `userId`, `message`, `conversationHistory`.
2. Fetch the user's current `health_profiles` document fresh from MongoDB (never
   stale/cached) — same discipline as the main scan pipeline.
3. Construct the chatbot system prompt (§3.4) with the profile interpolated in, plus
   the conversation history and new message.
4. Call MiniMax M3 via NVIDIA NIM using the exact request shape below.
5. Return the model's reply plus a short disclaimer string (see §3.5) alongside it —
   the disclaimer is a fixed string appended by the backend, not generated by the model.

### 3.3 Exact NVIDIA NIM Request Shape (use exactly this, adapted to Node/Express)

```python
# Reference shape (Python, as provided) — implement equivalently in Node/Express
# using fetch/axios with the same headers/payload structure.

import requests

invoke_url = "https://integrate.api.nvidia.com/v1/chat/completions"
headers = {
    "Authorization": "Bearer $NVIDIA_API_KEY",
    "Accept": "application/json"
}
payload = {
    "model": "minimaxai/minimax-m3",
    "messages": [
        {"role": "system", "content": "<chatbot system prompt from §3.4>"},
        # ...conversationHistory entries...
        {"role": "user", "content": "<new user message>"}
    ],
    "max_tokens": 1024,
    "temperature": 0.4,
    "top_p": 0.9,
    "stream": False
}
response = requests.post(invoke_url, headers=headers, json=payload)
```

**Node/Express equivalent notes:**
- `NVIDIA_API_KEY` must be read from an environment variable server-side, never
  exposed to the frontend.
- `max_tokens: 1024` (not 8192) — chatbot replies must stay short for speed; this is a
  quick-response feature, not a long-form generation feature.
- `temperature: 0.4` (not 1.00) — lower temperature for more consistent, grounded
  health-related answers rather than creative variation.
- `stream: false` for MVP simplicity, but consider `stream: true` with
  server-sent-events passthrough to the frontend if time permits, since streaming
  materially improves perceived speed for a "fast response" chatbot — this is the
  single highest-value optimization for the "max possible efficiency" goal, implement
  it if at all feasible.

### 3.4 Chatbot System Prompt

```
You are the Label-X health assistant, a fast, conversational alternative to the full
product scanning flow for Indian food/nutrition questions. You are speaking with a
user who has the following health profile:

Medical Conditions: {{CONDITIONS}}
Allergies: {{ALLERGIES}}
Dietary Preferences: {{DIETARY_PREFERENCES}}
Goals: {{GOALS}}
Additional Notes: {{ADDITIONAL_NOTES}}

Answer the user's question about food, ingredients, or nutrition claims, personalized
to their specific profile above — reference their actual conditions/allergies/goals by
name where relevant, not generic advice.

You do NOT have access to the user's actual product images or nutrition panels unless
they describe them to you directly in this conversation. If the user asks about a
specific packaged product without providing its ingredients or nutrition facts, ask
them to either describe the label or use the full Scan feature for an accurate,
image-based analysis — do not guess a specific product's nutrition facts from its name
alone.

Keep answers short and conversational — 2-4 sentences typically, longer only if the
question genuinely requires more detail. Never state or imply a medical diagnosis.
If the user's question involves a genuine medical emergency or symptom (not a
nutrition/food question), tell them to seek immediate medical attention rather than
answering as a nutrition assistant.

Do not fabricate specific regulatory citations (e.g. FSSAI rule numbers) unless you
are certain of them — if unsure, give general guidance and note that the full Scan
feature provides regulation-grounded claim checking.
```

Interpolate `{{CONDITIONS}}`, `{{ALLERGIES}}`, `{{DIETARY_PREFERENCES}}`,
`{{GOALS}}`, `{{ADDITIONAL_NOTES}}` directly from the fetched `health_profiles`
document fields (Part 3 §3) — verbatim field mapping, no renaming.

### 3.5 Required Disclaimer
Every chatbot response returned to the frontend must include a fixed disclaimer
string appended by the backend (not generated by the model, to guarantee it's always
present regardless of model output):

```
"disclaimer": "This is a quick estimate based on our conversation, not a full label scan. For a detailed, regulation-checked analysis, use the Scan feature."
```

Frontend must display this disclaimer visibly with every chatbot reply — small text
beneath the message bubble is sufficient, but it must not be omitted or hidden behind
a tooltip/expand action.

### 3.6 Chatbot UI Requirements (add to Part 4's screen list)
**Screen: Chatbot**
- Floating action button or persistent tab, accessible from Dashboard and Result page
  (so a user mid-scan-flow can still quickly ask a question)
- Standard chat bubble UI: user messages right-aligned, assistant messages
  left-aligned, disclaimer text small and muted beneath each assistant reply
- Loading state: a lightweight typing indicator (three-dot pulse), not the full
  multi-step scan loading modal — this must visibly feel faster than the scan flow
- Micro-animations (Framer Motion, per Part 4 §2 conventions): new messages slide/fade
  in on arrival; typing indicator uses a subtle pulsing/staggered dot animation

## 4. Translation — Purpose and Behavior

**Purpose:** allow scan results (and optionally chatbot replies) to be viewed in
Hindi, Telugu, Tamil, or other major Indian languages, without re-running the full
analysis pipeline per language.

**Model:** Gemini (same provider as the main scan pipeline — do not introduce a third
model for this).

### 4.1 Translation Route
| Route | Method | Purpose | Request Body | Response Body |
|---|---|---|---|---|
| `/api/translate` | POST | Translate a scan result's text fields into a target language | `{ scanId, targetLanguage }` | Translated text fields, same shape as source |

`targetLanguage` values: `"hi"` (Hindi), `"te"` (Telugu), `"ta"` (Tamil), extendable to
others as needed (e.g. `"bn"` Bengali, `"mr"` Marathi) — use standard ISO 639-1 codes.

### 4.2 What Gets Translated (and what doesn't)
**Translate these text fields only:**
- `personalized_verdict.summary`
- `claim_compliance[].reason`
- `quid_analysis[].statement`
- Key Risk Insight text (from the derived risk insights list)
- Proceed Anyway guidance text (`action` and `why` fields, if translation is requested
  for that view too)

**Never translate/leave unchanged:**
- Numeric fields (`risk_score`, percentages, nutrient gram values)
- `status` enum values (`non-compliant`, `needs-evidence`, etc.) — these are internal
  keys the frontend maps to its own localized badge labels, not raw model output
- `source` citation strings (regulation names stay in their original form, e.g. "FSSAI
  Advertising & Claims Regulations 2018" — do not translate legal/regulatory citations)

### 4.3 Backend Flow for `/api/translate`
1. Receive `scanId` and `targetLanguage`.
2. Check for an existing cached translation for this `scanId` + `targetLanguage`
   combination first (see §4.4 caching) — if found, return it directly, do not call
   Gemini again.
3. If not cached: fetch the original scan result via `scanId` → `scan_history` →
   `cached_results`.
4. Construct the translation prompt (§4.5) with the specific text fields listed in
   §4.2, and call Gemini.
5. Store the translated fields in a new cache entry, return to client.

### 4.4 Caching for Translations
Add to the schema (Part 3): a `translated_results` collection.

| Field | Type | Notes |
|---|---|---|
| `_id` | ObjectId | primary key |
| `cachedResultId` | ObjectId (ref: `cached_results`) | which product result this translation belongs to |
| `targetLanguage` | String | e.g. `"hi"`, `"te"`, `"ta"` |
| `translatedFields` | Object | same shape as the translatable fields listed in §4.2 |
| `createdAt` | Date | |

**Index:** `{ cachedResultId: 1, targetLanguage: 1 }` (compound, unique) — this means
translation is cached per PRODUCT (via its `cachedResultId`), not per individual scan
event, so if multiple users/scans hit the same cached product, translations are
reused across all of them too, not just within one user's repeat views.

### 4.5 Gemini Translation Prompt

```
Translate the following food-label analysis text fields into {{TARGET_LANGUAGE_NAME}}
({{TARGET_LANGUAGE_CODE}}). Keep the tone plain, clear, and accessible — this is
health information being read by an everyday consumer, not a technical or legal
translation.

Do NOT translate: numeric values, percentages, or the following exact terms, which
must remain in their original form: {{SOURCE_LIST}} (e.g. "FSSAI Advertising & Claims
Regulations 2018").

Text fields to translate:
{{FIELDS_JSON}}

Return ONLY a JSON object with the exact same keys as the input, with translated
string values, no extra commentary.
```

## 5. Definition of Done for This PRD
- `/api/chat` calls MiniMax M3 via NVIDIA NIM using the exact request shape in §3.3,
  with the user's fresh health profile interpolated into the system prompt.
- Every chatbot response includes the fixed disclaimer string from §3.5, appended
  server-side, always present regardless of model output.
- Chatbot `max_tokens` and `temperature` are set as specified (1024 / 0.4), not left at
  the original example's 8192 / 1.00 — those values are for general-purpose use, not
  tuned for a fast, grounded health chatbot.
- `/api/translate` uses Gemini, never MiniMax M3, and never re-runs the main scan
  analysis — it only translates already-generated text fields.
- Translation results are cached per `cachedResultId` + `targetLanguage` pair (§4.4) —
  requesting the same product's translation twice, even across different users, must
  not trigger a duplicate Gemini call.
- Numeric fields, status enums, and regulatory source citations are never passed
  through translation — verified by checking the translation prompt only receives the
  fields listed in §4.2.
- Chatbot UI (§3.6) is visibly and measurably faster-feeling than the full scan
  loading flow — typing indicator, not the multi-step modal.
